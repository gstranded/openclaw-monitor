# openclaw-monitor 第一阶段 MVP 范围与信息架构

> 对应 Issue: #13
> 
> 目标：为 openclaw-monitor 第一阶段提供可供前端、后端、QA 对齐的产品与页面基线。

## 1. 文档目标

本阶段只定义第一屏核心监控视图，确保团队先围绕统一页面结构、状态口径与事件表达建立共识，再进入实现。

页面聚焦四个区域：

- 顶部：Agent 状态卡片
- 中间：积分排行榜
- 左侧：活动时间线
- 右侧：实时事件流

## 2. MVP 目标

第一阶段 MVP 的目标不是覆盖所有 OpenClaw 管理能力，而是先让操作者能在一个页面里快速回答以下问题：

1. 现在有哪些 Agent 在工作、空闲、阻塞或异常？
2. 当前谁的积分最高，最近是否有明显变化？
3. 最近一段时间发生了哪些关键活动？
4. 此刻系统正在产生哪些实时事件，需要不需要关注？

## 3. In Scope

第一阶段明确纳入以下范围：

### 3.1 页面与模块

- 单页 Dashboard 首页
- 四栏核心布局：
  - 顶部 Agent 状态卡片区
  - 中间积分排行榜区
  - 左侧活动时间线区
  - 右侧实时事件流区
- 支持桌面端优先的信息浏览

### 3.2 Agent 状态卡片

每张卡片至少展示：

- Agent 名称
- 角色 / 职责
- 当前状态
- 最近活跃时间
- 当前任务摘要（若有）
- 当前积分（若可用）

### 3.3 积分排行榜

至少展示：

- Agent 名称
- 当前积分
- 排名
- 积分变化趋势（上涨 / 持平 / 下滑，可先用文本或图标表达）

### 3.4 活动时间线

至少展示最近关键活动：

- 任务开始
- 任务完成
- 提交 / PR / 回流
- 阻塞上报
- 验收结论

### 3.5 实时事件流

至少展示系统流式事件：

- Agent 心跳
- 状态切换
- 新任务接收
- 工具调用结果摘要
- 错误 / 阻塞 / 告警类事件

### 3.6 状态与术语基线

- 定义 Agent 状态口径
- 定义事件类型分层
- 定义页面空态、加载态、错误态
- 定义时间表达与排序规则

## 4. Out of Scope

第一阶段不纳入以下内容：

- 多页面复杂导航
- Agent 详情页
- 任务详情钻取页
- 自定义筛选器与高级搜索
- 用户权限体系
- 可编辑配置面板
- 告警规则配置
- 历史分析报表
- 移动端专项适配
- 完整设计系统建设
- 多组织 / 多项目视角切换
- 数据导出功能

## 5. 首页信息架构

## 5.1 页面层级

```text
Dashboard 首页
├── 顶部概览区
│   └── Agent 状态卡片列表
├── 主内容区
│   ├── 左侧：活动时间线
│   ├── 中间：积分排行榜
│   └── 右侧：实时事件流
└── 全局状态反馈
    ├── 加载态
    ├── 空态
    └── 错误态
```

## 5.2 阅读优先级

推荐用户阅读顺序：

1. 顶部 Agent 状态卡片：先判断整体运行健康度
2. 中间积分排行榜：再看当前产出表现
3. 左侧活动时间线：理解近期过程与上下文
4. 右侧实时事件流：监控当前动态变化

原因：

- 状态卡片回答“现在怎么样”
- 排行榜回答“谁在产出”
- 时间线回答“刚刚发生了什么”
- 事件流回答“此刻还在发生什么”

## 5.3 布局建议

- 顶部横向卡片区：展示 4~8 个 Agent 卡片
- 下半区三栏布局：
  - 左栏（活动时间线）：约 25%
  - 中栏（积分排行榜）：约 35%
  - 右栏（实时事件流）：约 40%

说明：实时事件流信息密度高，建议给予更宽区域。

## 6. 核心模块定义

## 6.1 顶部 Agent 状态卡片

### 目标

用最短时间告诉用户每个 Agent 是否正常工作。

### 最小字段

- `agentName`
- `role`
- `status`
- `lastActiveAt`
- `taskSummary`
- `score`

### 展示规则

- 按状态优先级排序：异常 / 阻塞 > 运行中 > 待命 > 离线
- 同状态下按最近活跃时间倒序
- 卡片颜色或标记需能快速区分状态

## 6.2 中间积分排行榜

### 目标

帮助操作者快速识别高产出 Agent 与近期变化。

### 最小字段

- `rank`
- `agentName`
- `score`
- `scoreDelta`
- `trend`

### 展示规则

- 默认按积分倒序
- 若积分相同，按最近更新时间倒序
- `scoreDelta` 为相对最近一个统计窗口的变化值

## 6.3 左侧活动时间线

### 目标

提供“近期发生了什么”的上下文，不要求完全实时，但要可追溯。

### 最小字段

- `timestamp`
- `agentName`
- `activityType`
- `summary`
- `linkedObject`（如 issue / PR / task，可选）

### 展示规则

- 按时间倒序
- 强调关键节点，不展示所有低价值噪音日志
- 可按活动类型加标签

## 6.4 右侧实时事件流

### 目标

提供“现在还在发生什么”的流式观察窗口。

### 最小字段

- `timestamp`
- `eventType`
- `level`
- `agentName`（若有）
- `message`

### 展示规则

- 默认按最新事件置顶或追加
- 优先展示 warning / error / blocked
- 高频事件可折叠或合并，但第一阶段先不强制实现聚合

## 7. 关键术语定义

## 7.1 Agent

在系统内承担特定职责、可接收任务并执行工作的智能体实例。

## 7.2 状态卡片

用于展示单个 Agent 当前概况的摘要卡片。

## 7.3 积分

用于反映 Agent 当前贡献或产出表现的数值指标。第一阶段只展示结果值，不在页面内解释完整计分公式。

## 7.4 活动

对任务进展有上下文意义的阶段性动作，例如开始任务、提交变更、完成验收、上报阻塞。

## 7.5 事件

系统运行过程中产生的实时信号，粒度通常比“活动”更细，可包含状态切换、心跳、错误、工具调用结果等。

## 8. 状态定义

## 8.1 Agent 状态枚举

建议第一阶段统一使用以下状态：

- `running`：正在处理任务，且近期有活跃信号
- `standby`：当前无执行任务，处于待命状态
- `blocked`：任务推进受阻，等待外部输入、权限、依赖或澄清
- `error`：执行过程中出现异常，需要关注
- `offline`：长时间无心跳或不可达

## 8.2 状态判定口径

### running

- 有当前任务
- 最近活跃时间在可接受窗口内
- 无已声明阻塞

### standby

- 无当前任务
- 可正常响应
- 无异常

### blocked

- Agent 已明确报告无法继续
- 阻塞原因应可读，如缺权限、缺接口、需求不清

### error

- 运行失败或出现未恢复异常
- 与 `blocked` 的区别是：`error` 偏系统或执行异常，`blocked` 偏业务推进受阻

### offline

- 超过约定时间未收到心跳或状态更新
- 第一阶段可由后端按固定阈值输出该状态

## 9. 页面状态定义

## 9.1 Loading

- 首次加载时展示骨架屏或占位态
- 四个核心模块都应有独立加载反馈

## 9.2 Empty

- 无 Agent 数据时：提示“暂无 Agent 数据”
- 无排行榜数据时：提示“暂无积分数据”
- 无活动时：提示“暂无近期活动”
- 无事件时：提示“暂无实时事件”

## 9.3 Error

- 模块级加载失败优先展示模块内错误提示
- 页面整体失败时展示整页错误态
- 错误提示至少包含：失败说明、重试入口（若有）

## 10. 时间与排序规则

- 默认使用统一时区展示（建议以后端标准输出为准）
- 所有列表默认按时间倒序
- 页面上使用相对时间 + 精确时间悬浮或副文案均可
- 时间线偏“阶段活动”，事件流偏“实时信号”

## 11. 推荐数据结构草案

```ts
interface AgentStatusCard {
  agentName: string
  role: string
  status: 'running' | 'standby' | 'blocked' | 'error' | 'offline'
  lastActiveAt: string
  taskSummary?: string
  score?: number
}

interface ScoreRankItem {
  rank: number
  agentName: string
  score: number
  scoreDelta?: number
  trend?: 'up' | 'flat' | 'down'
}

interface TimelineItem {
  timestamp: string
  agentName: string
  activityType: 'task_started' | 'task_completed' | 'commit' | 'pr_opened' | 'blocked' | 'qa_passed' | 'qa_failed'
  summary: string
  linkedObject?: {
    type: 'issue' | 'pr' | 'task'
    id: string
    url?: string
  }
}

interface EventStreamItem {
  timestamp: string
  eventType: string
  level: 'info' | 'warning' | 'error'
  agentName?: string
  message: string
}
```

## 12. 页面状态样例

## 12.1 首页有数据时

- 顶部至少展示 1 张 Agent 状态卡片
- 中间排行榜至少展示 1 条积分记录
- 左侧时间线至少展示 1 条近期活动
- 右侧事件流至少展示 1 条实时事件
- 页面不要求四个模块数据时间戳完全相同，但都应来自可解释的数据窗口

## 12.2 首页部分模块为空时

允许出现“部分模块有数据、部分模块为空”的情况，例如：

- 有 Agent 与积分数据，但暂无近期活动
- 有活动时间线，但事件流暂时没有实时信号
- 有 Agent 卡片，但某些 Agent 暂无积分

此时页面应：

- 保持整体布局稳定，不因空态打散区域结构
- 在模块内明确提示空原因或当前暂无数据
- 不把空态误展示为错误态

## 12.3 首页异常时

建议区分两类异常：

- 模块级异常：某一区域请求失败，但其他区域可正常展示
- 页面级异常：首页初始化依赖整体失败，无法完成基础渲染

对应展示规则：

- 模块级异常优先模块内提示，不阻断其他模块浏览
- 页面级异常展示整页错误说明，并保留重试入口
- 异常文案应尽量可审计，避免只显示“加载失败”而没有上下文

## 13. 前后端与 QA 对齐建议

### 前端

- 先按四区结构完成信息布局
- 先保证状态、空态、错误态一致
- 第一阶段可用静态 mock 数据对齐布局
- 建议先准备一组覆盖 running / standby / blocked / error / offline 的演示数据

### 后端

- 优先稳定输出四类模块最小字段
- 优先统一状态枚举与时间字段格式
- 事件流与时间线可共用底层事件源，但输出语义需区分
- 对 blocked / error / offline 提供清晰判定来源，避免前端二次猜测

### QA

- 按四区模块分别验证加载态、空态、错误态
- 验证状态口径是否一致
- 验证时间排序与字段缺省处理
- 验证 blocked / error / offline 的边界差异
- 验证“部分模块为空但页面可用”的场景
